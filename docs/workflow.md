# Workflow

Дисциплина работы с Claude Code: цикл задачи, контекст, антипаттерны.

## Рабочий цикл: Explore → Plan → Implement → Commit

**Explore.** Изучи код, пойми текущее состояние. Glob, Grep, Read. Ничего не меняй.

**Plan.** Переключись в plan mode (`Shift+Tab×2`). Покажи план пользователю. Жди одобрения. Только потом пиши код. Для сложных задач — Annotation Cycle: CC пишет `plan.md` → ты вставляешь заметки `> NOTE: ...` → CC адресует → повтор пока нет вопросов → "Implement the plan".

**Implement.** Код строго по плану. Если план оказался неверным — вернись в Plan, не импровизируй. `Shift+Tab×1` для auto-accept.

**Commit.** Коммит после каждой завершённой подзадачи. Коммит = save point. Английский, осмысленно: `feat: add hero section with stats bar`.

## Управление контекстом

**Реалии CC:** ~200K токенов, но ~20K съедает система. Качество начинает падать на 20–40% заполнения. Авто-compaction на ~83% теряет 70–80% деталей.

**Практический потолок: не давай контексту перевалить за 60%.**

**Правило одной задачи.** Одна сессия CC = одна задача (один spec). Закончил — новый чат. Свежие 200K дешевле, чем деградация.

**Document & Clear (лучше чем `/compact`):**
1. `Dump current progress, open decisions, and remaining tasks to handoff.md`
2. `/clear` — полный сброс
3. В новом чате: `Read handoff.md and continue from where we left off`

Ты контролируешь, что сохранится — в отличие от `/compact`.

**`/catchup` после `/clear`.** Команда настраивается в `specs/01a-local-setup.md`. После `/clear` → `/catchup` восстанавливает контекст по изменённым файлам ветки.

**Экономия токенов:**
- Ссылайся на файлы `docs/`, не инлайнь через `@file` в `CLAUDE.md`.
- Держи не больше 5–8 MCP-серверов — каждый добавляет схемы в контекст.
- `claude -c` — продолжить последнюю сессию, `claude -r` — вернуться к конкретной.

## Doom Loop и обработка ошибок

**Правило двух попыток.** Если CC не починил с двух попыток — стоп:
1. `/clear` — сброс контекста.
2. Переформулируй проблему с учётом того, что узнал.
3. Дай CC свежий контекст (KB-файлы + описание).

**Признаки, что пора бросить сессию:**
- CC повторяет одни и те же изменения по кругу.
- CC начинает менять код, который уже работал.
- CC предлагает «упростить», удалив функционал.
- CC говорит «pre-existing issue» или «out of scope» — это отмазки.

**Anti-rationalization.** Если CC заявляет, что задача выполнена, но ты видишь, что нет — скажи прямо: «This is not done. [Конкретно что не так]. Fix it.»

## Спецификации (`specs/`)

Каждая крупная задача = один файл в `specs/`, нумерация `00→13`. Шаблон — в `specs/templates/spec-template.md`. Полная последовательность — в `specs/INDEX.md`.

Минимальная структура спеки:
- **KB files to read first** — список `docs/*.md` (CC грузит только их, не всё)
- **Goal** — 1–2 предложения
- **Requirements** — что именно нужно
- **Tasks** — нумерованный список подзадач
- **Boundaries** — что можно/нельзя без подтверждения
- **Done when** — критерии готовности

Запуск спеки:
```
Read and execute specs/04-homepage-and-approval.md.
Before coding:
1. Read the KB files listed in the spec.
2. Show me your implementation plan.
3. Wait for my approval before writing any code.
```

## Верификация

- Давай CC тесты и команды для самопроверки — он работает драматически лучше, когда может проверить результат.
- Коммить **перед** большими изменениями.
- Writer/Reviewer паттерн: сессия 1 пишет код → сессия 2 (свежая) ревьюит.
- Один subagent = один deliverable, не план.

## Визуальные правки

CC не видит сайт — он видит код. Для визуальных правок:

1. Открой localhost, скриншот проблемного места.
2. Сохрани в `public/screenshots/fix-name.png` (или вне репо).
3. Скажи CC конкретно:
   ```
   Look at screenshot public/screenshots/fix-hero.png.
   Problems:
   - Headline is too small, needs to be 56px.
   - Bottom padding is only 20px, should be 80px.
   - CTA button blends with background, increase contrast.
   ```

**Главное правило:** конкретика > абстракция. Размеры в px, цвета в hex/класс, отступы в Tailwind-классах.

## Git-дисциплина

- Работай в ветке `dev`, в `main` попадаем только через PR — см. `docs/deploy.md`.
- Коммит после каждой подзадачи.
- Сообщения на английском, краткие.
- Не коммить: `.env`, `data/`, `node_modules/`, лог-файлы.
- Перед мержем в main — проверь на localhost.

## Hooks — гарантии вместо инструкций

Хуки работают на 100%, инструкции в `CLAUDE.md` — на ~70%. Готовый `.claude/settings.json` (защита `main`, блок `rm -rf`, автоформат) ставится один раз при инициализации проекта — см. `specs/01a-local-setup.md`.

## Антипаттерны — что не делать

- **Kitchen sink сессия.** Несколько разнородных задач в одном чате. → Одна задача = один чат.
- **Исправляю×5.** CC ошибся, опять ошибся, контекст отравлен. → После 2 попыток `/clear`.
- **Раздутый CLAUDE.md.** 200+ строк, CC игнорирует половину. → 60–80 строк, остальное в `docs/`.
- **«Сделай красиво».** Абстрактные запросы = абстрактный результат. → Конкретные размеры, цвета, px.
- **Вайбкодинг без плана.** 20 решений × 80% = 1% шанс, что всё правильно. → Plan mode всегда.
- **Забыл закоммитить.** CC сломал, откатить нельзя. → Коммит после каждой подзадачи.
- **Один чат на весь день.** К вечеру CC деградирует. → Новый чат на каждую задачу.
