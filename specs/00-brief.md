# Spec 00: Приём брифа и материалов проекта

## KB files to read first

- `.claude/memory/INDEX.md`

## Goal

Принять от пользователя готовые материалы предпроектной работы (бриф, анализ конкурентов, структуру страниц, тексты, медиа), разложить их в `docs/` и `public/`. Зафиксировать в памяти ключевые решения и табу заказчика. Подготовить почву для всех последующих спек.

## Background

Предпроектная работа (бриф клиента, SEO-анализ, написание структуры и текстов) идёт ВНЕ Claude — пользователь делает её сам с MCP (Keyso, JustMagic, Wordstat и др.) или вручную. На входе в Claude он приносит готовые `.md` файлы или экспорт со старого сайта (например, Tilda).

Этот спек — **точка приёма**, а не точка генерации. Если каких-то материалов нет — спросить пользователя, а не выдумывать.

## Входные форматы (что приносит пользователь)

Один из вариантов или их комбинация:

### Вариант A: Готовые .md файлы

Пользователь даёт пути к файлам:
- `brief.md` — бизнес, ЦА, услуги, бренд (цвета, тон), контакты, домен
- `competitors.md` — анализ ТОПовых конкурентов (URLs, что у них есть, чему учиться)
- `structure.md` или `pages.md` — древовидный список страниц
- `content/*.md` — тексты по страницам (один файл = одна страница)
- `media/` — папка с лого, фото, шрифтами

### Вариант B: Экспорт со старого сайта

- ZIP-выгрузка из Tilda
- URL действующего сайта для парсинга
- В этом случае → запустить опционально `specs/optional/opt-migrate-from-existing.md` параллельно

## Tasks

1. Спросить пользователя: какие материалы он передаёт, где они лежат
2. Прочитать все переданные файлы целиком
3. Создать `docs/spec.md` — выжимка из брифа: бизнес, ЦА, услуги, бренд (цвета HEX, шрифт, тон), контакты, домен, основные SEO-запросы
4. Создать `docs/content.md` — собранные тексты по страницам в едином формате (заголовок страницы → текст по секциям)
5. Создать `docs/pages.md` — структура страниц как Markdown-таблица:
   ```
   | URL | H1 | Title | Description | Приоритет | Статус | Родитель |
   ```
6. Создать `docs/integrations.md` — что планируется (CRM, аналитика, домен, внешние сервисы) — заполнено настолько, насколько известно сейчас, остальное по ходу
7. Скопировать медиа в `public/images/`, лого в `public/logo.svg` (если есть)
8. Записать в память:
   - `decisions.md` — ключевые решения брифа (URL-структура, выбор CRM, отказ от/выбор технологий)
   - `feedback.md` — табу и правила заказчика с **Why:** (нет фото юристов, не использовать слово X, бренд-цвета, и т.д.)
   - `references.md` — домен, планируемая CRM, контакты заказчика
   - `project_state.md` — отметить `00-brief` как done, следующая спека `01a-local-setup`
9. Показать пользователю summary: «Прочитал бриф. Понял: [3-5 ключевых пунктов]. Зафиксировал в памяти: [список]. Готов к 01a-local-setup. Что-то добавить/исправить?»

## Boundaries

- **Always:** читать переданные файлы целиком, задавать уточняющие вопросы при противоречиях
- **Ask first:** если в брифе нет цветов / контактов / домена — спросить, не выдумывать
- **Never:** создавать `docs/spec.md` из своей фантазии, генерировать тексты «вместо» пользователя, начинать кодить

## Done when

- `docs/spec.md`, `docs/content.md`, `docs/pages.md`, `docs/integrations.md` созданы и заполнены реальными данными
- Медиа разложено в `public/`
- Память обновлена: `decisions.md`, `feedback.md`, `references.md`, `project_state.md`
- Пользователь подтвердил summary

## Memory updates

- `decisions.md` — все не-очевидные решения брифа
- `feedback.md` — табу и правила (с **Why:**)
- `references.md` — домен, CRM, контакты, ID счётчиков (если уже известны)
- `project_state.md` — фаза «brief done, ready for infrastructure»
