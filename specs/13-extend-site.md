# Spec 13: Расширение сайта после релиза (post-launch loop)

## Когда применять

Сайт сдан заказчику (`12-handoff` выполнен), сайт работает в проде. Возникла задача:
- Добавить новую страницу (услуга, посадочная под рекламу, информационная)
- Добавить новый блок на существующую страницу
- Написать статью в блог
- Точечная правка контента/дизайна по запросу заказчика

Это не «новая разработка», а живой цикл сайта. Спека универсальная — применяется циклически на всём жизненном протяжении проекта.

## KB files to read first

- `.claude/memory/INDEX.md` (всегда сначала! поймёшь актуальный контекст)
- `.claude/memory/project_state.md` (где сейчас проект)
- `.claude/memory/feedback.md` (табу заказчика — не повторять старых ошибок)
- `.claude/memory/pointers.md` (где какой компонент — найти быстрее)
- docs/pages.md (если задача про страницы)
- docs/content-layout.md (если задача про новые секции)

## Goal

Внести изменение в production-сайт: качественно (без регрессий), быстро (используя готовые шаблоны и компоненты), документировано (память обновлена). Минимизировать риск сломать то что работало.

## Типы задач (выбрать сценарий)

### A. Новая страница

1. Если тип — стандартная страница услуги → использовать `ServicePageTemplate` (см. `pointers.md`):
   - Скопировать структуру из любой существующей страницы услуги
   - Создать `app/[slug]/page.tsx` + `page-data.ts`
   - Заполнить контентом из новой выдачи пользователя (или попросить тексты)
   - Добавить в `app/sitemap.ts`
   - Обновить `docs/pages.md` (статус: новая)
   - Перелинковать с существующими (Related, навигация если нужно)
   - Внутренние ссылки на эту страницу с релевантных существующих
2. Если кастомная — собрать из секций `components/sections/*` по принципу главной
3. Использовать шаблон `specs/templates/page-spec-template.md` для записи требований (если задача сложная)

### B. Новый блок на существующей странице

4. Определить тип секции из `docs/content-layout.md` (там 44 типа)
5. Проверить `pointers.md` — может уже есть готовый компонент
6. Если есть — переиспользовать, добавить вызов в нужную страницу
7. Если нет — создать `components/sections/[NewSection].tsx` (server component если возможно)
8. Помнить про порядок секций (см. шаблон страницы) — куда логично встроить

### C. Статья в блог

9. Создать `content/blog/[slug].mdx` с frontmatter (см. шаблон в спеке 07)
10. Вставить контент пользователя
11. Если первая статья за долгое время — проверить что блог-инфраструктура актуальна (не сломалось ли при последнем обновлении Next)
12. Опционально: создать перелинковку из новой статьи на коммерческие страницы (CTA в конце статьи)

### D. Точечная правка контента/дизайна

13. Прочитать запрос заказчика дословно
14. Понять scope: одна страница / все страницы / конкретный компонент
15. Если скоупа касается переиспользуемого компонента — предупредить, что изменение применится везде
16. Сделать правку, проверить локально, задеплоить

## Универсальные шаги (для всех сценариев)

17. **Перед началом:** `git tag pre-extend-[YYYY-MM-DD]` — откат
18. **Делать в plan mode:** показать пользователю что планируешь до правок
19. **Коммит после каждой завершённой подзадачи** — save point
20. **Локальная проверка:** `npm run dev`, посмотреть результат глазами
21. **Build-проверка:** `npm run build` — нет ошибок
22. **Деплой по схеме:**
    - **A:** `npm run build && pm2 restart [project]`
    - **B:** `git push origin dev` → ревью → merge в main → автодеплой
23. **Прод-проверка:** открыть страницу, проверить результат, проверить что соседние страницы не сломались
24. **Обновить память:**
    - `project_state.md` — что добавлено
    - `pointers.md` — если создан новый переиспользуемый компонент
    - `decisions.md` — если решение неочевидное
    - `feedback.md` — если по ходу заказчик дал новые корректировки

## Boundaries

- **Always:** plan mode перед изменениями, коммит после каждой подзадачи, локальная проверка перед prod-деплоем
- **Ask first:** перед массовыми правками (затрагивает > 5 страниц), перед обновлением зависимостей (npm update), перед изменением общих компонентов (Header, Footer, ServicePageTemplate)
- **Never:** делать «заодно ещё рефакторну» — задача в скоупе, расширения отдельной задачей; править прод напрямую без коммита; пушить в `main` напрямую (защищено protected branch — только через PR из `dev`)

## Когда отказаться и сделать новой большой спекой

Если запрос заказчика на самом деле:
- **Редизайн** (новая дизайн-система) — это не extend, это новая спека `03-design-system v2`
- **Новая большая фича** (личный кабинет, мультиязычность, e-commerce) — отдельная спека (см. `optional/`)
- **Полный SEO-переаудит** — отдельная спека `08-seo-schema v2`
- **Полный performance-переаудит** — повторить `11-performance` целиком

Сказать заказчику честно: «Это больше чем правка, я предлагаю отдельную спеку с планом и оценкой».

## Done when

- Изменение в проде, работает корректно
- `npm run build` проходит, console чист
- Соседние страницы не сломались (быстрая проверка на 2-3 главных)
- Память обновлена согласно сделанному
- Заказчик подтвердил что задача решена

## Memory updates

- `project_state.md` — описание правки + дата
- `pointers.md` — новые переиспользуемые элементы
- `feedback.md` — новые корректировки от заказчика
- `decisions.md` — нестандартные решения
- `lessons.md` — если что-то сломалось и починили

## Совет по работе с памятью

После 5-10 итераций `13-extend-site` память может разрастись. **Раз в квартал** — пересмотреть `project_state.md`: убрать историю сильно старых правок (они в git), оставить актуальное состояние. `feedback.md` — наоборот, не сокращать (правила заказчика — навсегда).
