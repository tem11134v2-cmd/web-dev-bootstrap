# Spec [opt]: Миграция со старого сайта (Tilda и др.)

## Когда применять

У заказчика есть действующий сайт (Tilda, WordPress, самописный, любой), который заменяется новым. Нужно сохранить SEO-позиции, контент, медиа.

**Применять параллельно/перед** `00-brief.md`. Часто экспорт со старого сайта — это и есть бриф (контент уже есть, нужно его извлечь и переложить).

## KB files to read first

- docs/seo.md (раздел про редиректы 301, перелинковку)
- docs/pages.md (если уже есть — обновим)

## Goal

Извлечь со старого сайта всё ценное (тексты, медиа, шрифты, цвета, структуру URL), сохранить SEO-наследие через 301-редиректы, разложить материалы в формат, готовый к работе по основным спекам.

## Background

Заказчик может прийти с одним из вариантов:
- **Tilda export** — ZIP-архив выгрузки (HTML + ассеты)
- **URL действующего сайта** — нужно парсить
- **CMS-доступ** (WordPress админка) — экспорт через плагин
- **Скриншоты + текстовый файл** — если нет технического доступа

## Tasks

### 1. Определить источник и план

1. Спросить пользователя:
   - Что есть на руках? (zip / URL / админка / только скриншоты)
   - Что нужно сохранить? (контент, дизайн, URL, медиа — обычно всё)
   - Что менять? (стек, дизайн, тон — обсудить scope)
2. Собрать список всех URL текущего сайта:
   - Из sitemap.xml старого сайта (если есть)
   - Через краулер: `wget -r --spider [URL] 2>&1 | grep '^--' | awk '{print $3}' | sort -u`
   - Из Я. Вебмастера старого сайта (раздел «Страницы в поиске»)

### 2. Экстракция активов

#### Если Tilda ZIP:

3. Распаковать архив
4. Извлечь:
   - **Изображения:** `images/` папка в Tilda export → переложить в `public/images/migrated/`
   - **Шрифты:** обычно в `css/` ссылки на CDN (Google Fonts или Tilda CDN) → определить семейство, добавить через `next/font`
   - **Логотип:** найти SVG/PNG лого → `public/logo.svg`
   - **Цвета:** парсить inline `style=""` атрибуты или CSS — собрать палитру
   - **Тексты:** из HTML файлов через парсер (см. ниже)

#### Если URL действующего сайта:

5. Краулинг через скрипт на Node:
   ```bash
   npx playwright install chromium  # если нужна интерактивность
   # или просто wget/curl + cheerio для парсинга HTML
   ```
6. Скачать все страницы как HTML + извлечь:
   - Тексты по секциям
   - URL изображений → скачать в `public/images/migrated/`
   - Метаданные (title, description, og)

#### Если только скриншоты + текстовый файл:

7. Тексты — пользователь даёт `.md` напрямую (это уже бриф)
8. Дизайн — фотографируем со скриншотов, реализуем в новой системе

### 3. Разложить материалы

9. Создать `docs/spec.md` (если ещё нет) с цветами, шрифтами, лого, контактами — извлечёнными со старого сайта
10. Создать `docs/content.md` с текстами по страницам:
    ```markdown
    ## Страница: /eb1/
    ### Hero
    Заголовок: ...
    Подзаголовок: ...

    ### Описание услуги
    ...
    ```
11. Создать/обновить `docs/pages.md` — все URL старого сайта в таблице:
    ```
    | Старый URL | Новый URL | H1 | Title | Статус |
    |-----------|-----------|----|----|--------|
    | /viza-o1/ | /o1/      | ... | ... | migrate |
    ```
12. Список изображений в `public/images/migrated/` — пересмотреть, удалить мусор, переименовать осмысленно

### 4. Карта редиректов 301 (САМОЕ ВАЖНОЕ для SEO)

13. Для **каждого** URL старого сайта определить новый URL:
    - URL не меняется → не редирект
    - URL меняется → 301 на новый
    - Страница удаляется → 301 на ближайший родственник (не на главную!)
    - Категория схлопнулась → 301 на новую категорию
14. Записать все редиректы в `docs/pages.md → раздел Редиректы`
15. В спеке `08-seo-schema.md` они попадут в `next.config.ts` и/или nginx
16. **Минимум потерь SEO:** внутренние ссылки нового сайта должны вести на конечные новые URL (не на старые с редиректом)

### 5. Сохранение позиций

17. Перед запуском нового сайта:
    - Снять текущие позиции в Я.Вебмастере (раздел «Поисковые запросы»)
    - Снять GSC (Performance → Pages, последние 28 дней)
    - Сохранить как baseline в `.claude/memory/lessons.md`
18. После запуска — мониторить просадку позиций 2-4 недели:
    - Если критичная просадка (>30% по основным запросам) — проверить редиректы (нет ли 404, нет ли цепочек)
    - Сообщить в Я.Вебмастер о смене URL через инструмент «Переезд сайта» (если меняется домен или структура)

### 6. Запуск (cutover)

19. Заглушка/старый сайт работает до момента запуска нового
20. После одобрения нового (спека 04 и далее) — резкое переключение DNS или замена nginx upstream
21. Сразу после переключения:
    - Проверить главные страницы (нет 502, нет 404)
    - Прогнать все старые URL — попадают ли на новые (curl -I, ожидаем 301)
    - Sitemap нового сайта отправлен в Я. Вебмастер и GSC

## Boundaries

- **Always:** сохранять SEO-наследие через 301, бэкап старого сайта перед удалением, baseline позиций до запуска
- **Ask first:** перед удалением любых страниц старого сайта (вдруг по ним идёт трафик), перед сменой структуры URL без редиректов
- **Never:** делать 302 (временный) вместо 301 (постоянный) — позиции не передаются, удалять старый сайт раньше чем новый стабильно заработает

## Done when

- Все материалы старого сайта извлечены в `public/`, `docs/spec.md`, `docs/content.md`
- `docs/pages.md` содержит полную карту старых → новых URL с редиректами
- Изображения оптимизированы (см. спека 11)
- Baseline позиций сохранён в `lessons.md`
- Готово к работе по основному потоку (можно сразу запускать `02-project-init`, минуя `00-brief` если данные собраны)

## Memory updates

- `decisions.md` — какие URL остались, какие изменились, почему (часто заказчик хочет поменять — обсудить SEO-цену)
- `references.md` — где лежит бэкап старого сайта, baseline позиций, доступ к Я. Вебмастер старого сайта
- `lessons.md` — baseline позиций ДО запуска (для сравнения)
- `project_state.md` — отметка «миграция со старого сайта проведена, готово к [02-project-init]»
