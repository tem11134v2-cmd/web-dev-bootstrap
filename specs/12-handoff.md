# Spec 12: Передача проекта заказчику + runbook

## KB files to read first

- docs/deploy.md (раздел «Собственность», «Откат прода»)
- docs/server-manual-setup.md (раздел «Обслуживание», «Типовые проблемы»)
- docs/server-add-site.md (раздел «Частые проблемы»)
- `.claude/memory/references.md` (всё что заказчик должен получить)
- `.claude/memory/decisions.md` (что заказчик должен знать о решениях)

## Goal

Передать заказчику все доступы и инструкции для самостоятельной эксплуатации сайта. Создать короткий runbook для типичных аварий — чтобы заказчик-нетехнарь пережил ночь без разработчика. Зафиксировать момент передачи.

## Tasks

### 1. Передача доступов

1. Убедиться что у заказчика есть/получены:
   - **Anthropic аккаунт** (для Claude Code) — свой, не разработчика
   - **GitHub аккаунт + ownership репозитория.** Разработчик — collaborator, может быть удалён
   - **VPS root SSH доступ** (или хотя бы доступ к панели управления VPS)
   - **Домен** в личном реестре регистратора
   - **Cloudflare аккаунт** (если используется)
   - **CRM, Метрика, GA, GSC, Я.Вебмастер** — все аккаунты на заказчике
2. Если что-то на разработчике — перенести (смена owner GitHub-репо, передача SSH-доступа на VPS)

### 2. Создать handoff-документ для заказчика

3. Создать `HANDOFF.md` в корне проекта (русский, простыми словами) с разделами:
   - **Что у вас есть:** домен, VPS, репо, аккаунты — со ссылками
   - **Как открыть сайт:** просто URL
   - **Как править контент:** через Claude Code на VPS — пошагово
   - **Куда смотреть статистику:** ссылки на Метрику/GA/Вебмастер
   - **Куда приходят заявки:** CRM URL + ответственный
   - **Кому писать если что-то сломалось:** разработчик (контакты) + сроки реакции
4. В `HANDOFF.md` — раздел **Runbook** (см. ниже)

### 3. Runbook аварий (внутри HANDOFF.md)

Однострочные команды для типовых проблем:

```markdown
## Сайт не открывается (502 Bad Gateway)

ssh deploy@[ip]
pm2 logs [project]-prod --lines 50  # посмотреть ошибки
pm2 restart [project]-prod          # перезапустить

## SSL-сертификат истёк

ssh deploy@[ip]
sudo certbot renew                  # обычно auto, но можно вручную
sudo systemctl reload nginx

## Сайт работает, но обновления не появились

# Проверить GitHub Actions: Settings → Actions → последний run
# Если упал — кликнуть "Re-run failed jobs"
# Если не запустился — проверить что merge dev→main произошёл и защищённая ветка не блокирует
# В крайнем случае — ручной pull на сервере:
ssh deploy@[ip]
cd ~/prod/[project]
git pull origin main && npm ci && npm run build && pm2 restart [project]-prod

## Билд падает с OOM (out of memory)

ssh deploy@[ip]
free -h                             # проверить swap
# Если swap < 2 GB — добавить:
sudo fallocate -l 2G /swapfile && sudo chmod 600 /swapfile
sudo mkswap /swapfile && sudo swapon /swapfile

## Откат на последнюю рабочую версию

ssh deploy@[ip]
cd ~/prod/[project]
git log --oneline -10               # найти последний рабочий коммит
git reset --hard [commit-hash]
npm ci && npm run build && pm2 restart [project]-prod

## Лиды не доходят в CRM

# 1. Проверить fallback: ssh deploy@[ip] && tail data/leads.json
# 2. Если лиды есть в leads.json но нет в CRM — токен/webhook слетел
# 3. Обновить .env (AMO_CRM_TOKEN или BITRIX_WEBHOOK_URL)
# 4. pm2 restart [project]-prod

## Сайт работает медленно

# Запустить PSI: pagespeed.web.dev → ввести URL → Mobile + Desktop
# Если упало после изменений — git revert последнего коммита
# Если деградировало без изменений — возможно VPS перегружен:
ssh deploy@[ip]
htop                                # CPU/RAM
df -h                               # свободное место
```

### 4. Регулярное обслуживание

5. В `HANDOFF.md` — раздел «Раз в месяц»:
   - `apt update && apt upgrade` на VPS
   - `npm outdated` в проекте + обновление зависимостей
   - Проверка `df -h` (свободное место)
   - Проверка `certbot certificates` (SSL не истекает)
   - Просмотр Я.Вебмастер на ошибки индексации
   - Просмотр PSI — нет ли деградации

### 5. Отзыв прав разработчика (если применимо)

6. Если разработчик уходит:
   - Удалить SSH-ключ разработчика из `~/.ssh/authorized_keys` на VPS
   - Убрать из collaborators GitHub-репо
   - Сбросить пароль deploy-пользователя (опционально)
7. Это инструкция для заказчика, в `HANDOFF.md` отдельный раздел

### 6. Финальные коммиты и теги

8. Коммит `chore: add HANDOFF.md`
9. Тег `v1.0` — релизная версия
10. Push тег в GitHub: `git push origin v1.0` (сам коммит должен быть уже в `main` через PR).

### 7. Передача документации

11. Заказчику отправить:
    - Ссылку на `HANDOFF.md` в репо
    - Список всех ID/URL из `references.md` (можно скопировать в .md и отправить)
    - Контакты для вопросов
12. Провести короткую видео-встречу или Loom-видео (10-15 мин), показать:
    - Где статистика, где CRM, где сайт
    - Как открыть GitHub-репо, как запустить Claude Desktop локально и попросить правки
    - Что делать если упало

### 8. Зафиксировать передачу

13. В `.claude/memory/project_state.md`:
    - Фаза «handed off, [дата]»
    - Дальше — `13-extend-site` для расширений или новые спеки по запросам заказчика

## Boundaries

- **Always:** убедиться что заказчик владеет ВСЕМ (домен, VPS, аккаунты), документировать каждый креденшел в `references.md`
- **Ask first:** перед удалением своих доступов (вдруг останутся хвосты) — обсудить с заказчиком timing
- **Never:** оставлять секреты на своей машине после передачи, удалять `.claude/memory/` (это часть проекта, переезжает)

## Done when

- Заказчик владеет всем (домен, VPS, GitHub, аккаунты)
- `HANDOFF.md` создан, содержит runbook + регулярное обслуживание + отзыв прав
- Проведён созвон или записан Loom (опционально, но желательно)
- Тег `v1.0` создан
- `project_state.md` зафиксировал момент передачи

## Memory updates

- `project_state.md` — фаза «handed off [YYYY-MM-DD]», ready for `13-extend-site`
- `references.md` — финальная сверка, всё актуально
- `lessons.md` — что вынесли из проекта в целом (если есть универсальные уроки — сохранить также в user-memory разработчика)
